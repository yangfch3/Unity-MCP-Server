## 2. 自由评审意见

1. **GetGameObjectPath 逻辑复用方式未明确**。设计文档说"复用 HierarchyTool 中的同一逻辑"，但 HierarchyTool 中的 `GetGameObjectPath` 是 `private static` 方法。实现时需要决定：是提取为共享 helper（如 `GameObjectUtils.GetGameObjectPath`），还是在 FindGameObjectsTool 中复制一份。建议在设计文档中明确复用策略，避免实现时引入不必要的代码重复或架构变动。

2. **WildcardMatch 方法的可测试性设计**。Testing Strategy 中提到"可从工具类中提取为 internal static，便于直接测试"，这是一个好决策，但在 Components and Interfaces 部分未体现——当前描述的所有辅助方法都没有标注可见性。建议在设计中明确 `WildcardMatch` 为 `internal static`，保持设计文档与测试策略的一致性。

3. **Correctness Properties 的覆盖度很高**。7 个属性覆盖了过滤正确性、等价性、交集语义、截断准确性、结构完整性和错误拒绝，形成了严谨的验证网络。Property 3/4 由于 EditMode 限制选择用具体示例覆盖的决策也合理。

4. **InputSchema 缺少 `anyOf` / `required` 约束**。JSON Schema 中没有表达"namePattern 和 componentType 至少提供一个"的约束。虽然这个校验在 Execute 方法中做了，但如果 MCP 客户端支持 Schema 级别的参数校验，缺少 `anyOf` 约束可能导致客户端无法在发送前提示用户。不过考虑到目前 MCP 生态中 Schema 校验能力有限，这不是关键问题。

## 3. 改进建议

### 🔴 关键问题（必须修改）

（无）

### 🟡 建议改进（推荐修改）

1. **补充 Out of Scope 段落** — 建议在 Overview 之后增加明确的 Out of Scope 说明，至少覆盖：不支持正则表达式、不区分 active/inactive GO（搜索包含所有）、不支持跨场景搜索。这能减少实现时的歧义和未来的需求蔓延。

2. **统一无匹配时的输出 JSON 结构** — 当前三种输出示例（正常/截断/无匹配）的字段集不一致。建议无匹配时也包含 `truncated: false, totalFound: 0`，保持所有成功响应的字段集一致，降低 Agent 解析复杂度。

3. **明确 GetGameObjectPath 的复用策略** — 在 Components and Interfaces 的"与现有组件的关系"中补充一句：是将 HierarchyTool 的 `GetGameObjectPath` 提取为共享 internal static 方法，还是独立实现一份。推荐前者以避免逻辑重复。

4. **明确空字符串 namePattern 的行为** — 在参数校验逻辑或 Data Models 中补充：`namePattern` 为空字符串时视为"未提供"（等同于 null），与 componentType 的空字符串处理保持一致。

### 🟢 锦上添花（可选优化）

1. **补充大场景性能说明** — 在 Architecture 或 Error Handling 部分增加一句对性能的预期说明，如"当前版本针对中小场景（<5000 GO）设计，大场景的性能优化（如提前终止遍历）留作后续迭代"。这能帮助使用者和未来维护者理解设计边界。

2. **标注 WildcardMatch 的可见性为 internal static** — 在 Components and Interfaces 的辅助方法列表中明确标注 `WildcardMatch` 为 `internal static`，与 Testing Strategy 中的提取策略保持一致。
