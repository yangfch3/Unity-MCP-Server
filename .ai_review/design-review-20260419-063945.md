## 2. 自由评审意见

1. **现有代码中 `HierarchyTool` 无 Prefab Stage 逻辑**：当前 `HierarchyTool.cs` 完全不引用 `PrefabStageUtility`，设计文档将"Prefab Stage 优先"描述为缺省行为的一部分，但这实际上是**新增行为**，而非"保持不变"。设计文档 Overview 中"缺省/空串 → Prefab Stage 优先，回退 Active Scene"容易让人误解这是已有行为。建议在文档中明确标注这是本次引入的行为变更。

2. **`SelectionTool`（`editor_getSelection`）与 `SelectGameObjectTool`（`editor_selectGameObject`）的命名对称性**：现有 `SelectionTool` 是"获取选中"，新增工具是"设置选中"。命名上 `SelectionTool` vs `SelectGameObjectTool` 不太对称，但考虑到 MCP 工具名（`editor_getSelection` / `editor_selectGameObject`）已经在需求中定义，且语义上也清晰，此处只是一个观察，不构成问题。

3. **`SearchInRoot` 路径匹配策略**：设计中 `SearchInRoot` 要求 `segments[0] == root.name`，意味着路径必须以根 GO 的名称开头（如 `"Root/Child"`）。同时 `FindByPath` 中做了 `TrimStart('/')`，所以 `"/Root/Child"` 和 `"Root/Child"` 等价。这与 `SelectionTool.GetGameObjectPath` 输出的路径格式（带前导 `/`）兼容。但如果用户传入的路径来自其他来源（如手动拼接），可能会踩到 `"//Root/Child"` 之类的边界。建议在 Error Handling 中补充路径格式相关的说明或做 normalize。

4. **属性测试依赖 FsCheck 的可行性**：Testing Strategy 提到"NUnit + FsCheck（或手动随机生成）"，但项目当前未引入 FsCheck 依赖。建议明确选择哪种方案，或在设计文档中注明需要新增依赖。

## 3. 改进建议

### 🔴 关键问题（必须修改）

1. **输出模型与伪代码不一致** — Data Models 中 `SelectGameObjectTool` 输出包含 `instanceID` 字段，但 Components 伪代码中 `Execute` 返回的是 `ToolResult.Success(name + path 信息)`，未提及 `instanceID`。需统一：要么伪代码补上 `instanceID`，要么输出模型删除该字段。

### 🟡 建议改进（推荐修改）

1. **补充 Out of Scope 声明** — 建议在 Overview 后新增一个 Out of Scope 小节，显式排除：多选支持（`Selection.gameObjects` 多对象）、InstanceID 定位、DontDestroyOnLoad 场景直接遍历等。

2. **标注 Prefab Stage 行为为新增** — Overview 的"核心变更"第 1 条建议改为："新增 `root` 参数路由逻辑，**同时将缺省模式从仅遍历 Active Scene 改为 Prefab Stage 优先**"，使行为变更一目了然。

3. **注明 PrefabStageUtility 命名空间版本差异** — 在 Components 或 Architecture 中注明：Unity 2018.3–2020.x 使用 `UnityEditor.Experimental.SceneManagement.PrefabStageUtility`，Unity 2021+ 使用 `UnityEditor.SceneManagement.PrefabStageUtility`。或注明项目最低支持的 Unity 版本。

4. **路径 normalize 增强** — `FindByPath` 中仅做了 `TrimStart('/')`，建议补充对连续 `/`、尾部 `/` 的处理说明，或在 Error Handling 表中增加"路径格式异常"的条目。

### 🟢 锦上添花（可选优化）

1. **Overview 补充动机** — 在 Overview 开头用一两句话说明"为什么要做"（从需求文档的 Introduction 中提炼），减少读者需要跨文件查阅的认知负担。

2. **明确属性测试方案** — Testing Strategy 中"NUnit + FsCheck（或手动随机生成）"建议选定一种方案，避免实现时的选择困难。考虑到项目当前无 FsCheck 依赖，建议默认使用手动随机生成 + System.Random，并在注释中说明。
