## 🔴 关键问题（必须修改）

### 1. MaxBufferSize 为 2500，设计文档和需求要求 5000

**位置**：`ConsoleTool.cs:21`

```csharp
private const int MaxBufferSize = 2500; // 实际
// design.md: MaxBufferSize: 1000 → 5000
// requirements.md Req 6.1: maximum capacity of 5000
```

代码实现为 2500，但 design.md 和 requirements.md 均写 5000。测试代码（Property 4）也跟着硬编码了 2500，导致测试绿色但与设计不符。

**处理方式二选一**：
- (a) 将 MaxBufferSize 改为 5000，同步更新 Property 4 测试
- (b) 如果 2500 是有意降级（如性能考量），更新 design.md 和 requirements.md

### 2. index 字段在 FIFO 淘汰后不稳定

**位置**：`ConsoleTool.cs:93-94` / `ConsoleTool.cs:136-137`

返回 JSON 中的 `"index"` 值使用的是 `_buffer` 的数组下标 `idx`。由于 `RemoveAt(0)` 会使所有元素前移，同一条日志在不同时间点查询会返回不同的 index 值。

**场景**：
1. buffer 满 2500 条，用户查到 index=2000 处有一条 Error
2. 新日志进来，RemoveAt(0) 执行，原 index=2000 的日志变成 index=1999
3. 用户用 `beforeIndex=2000` 查上下文，指向了错误的日志

设计文档描述的是「Index 字段在日志写入缓冲区时赋值」，暗示 index 应是全局递增的稳定 ID，而非可变的数组下标。

**建议**：引入一个 `_nextIndex` 递增计数器，写入时 `entry.Index = _nextIndex++`，查询时用这个稳定 ID。`beforeIndex` 参数查找时做 binary search 或 O(1) 偏移计算。

## ⚠️ 建议改进（推荐修改）

### 3. AssetDeleteFolderTool 删除 .meta 文件是设计文档外的额外行为

**位置**：`AssetDeleteFolderTool.cs:53-56`

```csharp
string metaPath = fullPath + ".meta";
if (File.Exists(metaPath))
    File.Delete(metaPath);
```

这是合理的 Unity 实践（不删 .meta 会导致 AssetDatabase 残留引用），但 design.md 伪码和 requirements.md 均未提及。

**建议**：保留代码，反向更新 design.md 的伪码，在 `Directory.Delete` 后加上 `.meta` 清理步骤。

### 4. ValidatePath 对 symlink 未做特殊处理

**位置**：`AssetDeleteFolderTool.cs:72-86`

`Path.GetFullPath` 只解析逻辑路径（`..` 等），不解析 symlink。如果 `Assets/SomeLink` 是指向项目外的 symlink，ValidatePath 会返回通过（路径前缀匹配 Assets），但 `Directory.Delete` 实际删除的是外部目录内容。

design.md Property 1 已提及 symlink 场景，但代码未处理。

**建议**：在 Unix 环境下，对 `fullPath` 调用 `new DirectoryInfo(fullPath).ResolveLinkTarget(true)` 或 P/Invoke `realpath` 做一次真实路径解析后再检查前缀。如果认为此风险可接受（内部工具、受信任调用方），至少在代码注释中标注这一已知限制。

### 5. ParseEntries 辅助方法在两个测试文件中完全重复

**位置**：`ConsoleToolTests.cs:31-62` 与 `ConsoleToolPropertyTests.cs:19-65`

`ParseEntries`、`ExtractStringField` 方法逐行相同。

**建议**：提取到 `Tests/Editor/TestHelper.cs`（或 `ConsoleToolTestHelper.cs`）中作为 `internal static` 方法，两个测试类引用。

### 6. ConsoleTool.cs 在 lock 块内 return

**位置**：`ConsoleTool.cs:76`

```csharp
lock (_lock)
{
    if (beforeIndex >= _buffer.Count)
        return Task.FromResult(ToolResult.Error(...)); // lock 内 return
```

C# 的 `lock` 语句在 return 前会正确释放锁，这不是 bug。但 lock 块内包含 early return 是容易误导后续维护者的模式。

**建议**：将越界检查提到 lock 外（先获取 count 快照再判断），或在注释中标注 `lock` 的 return 行为。

## ℹ️ 轻微备注

### 7. Property 1 只验证了 ValidatePath 纯函数

`AssetDeleteFolderToolPropertyTests.Property1_PathSafetyValidation` 直接调用 `ValidatePath()` 而非 `Execute()`。这是更精确的隔离测试，但不是端到端验证。鉴于 `Execute` 中 ValidatePath 之后还有 `Directory.Exists` 守卫，安全性保障链路完整，此处可接受。

### 8. InputSchema 为超长单行字符串

`ConsoleTool.cs:19` InputSchema 增加参数后变成约 350 字符的单行字符串。功能正确但可读性差。可考虑用 `@""` 多行字符串或 const 拼接提升可读性（非阻塞）。
