## 2. 自由评审意见

1. **MainThreadQueue 调度职责不清**：当前 `JsonRpcDispatcher.HandleToolsCall` 已将所有工具执行通过 `_mainThreadQueue.Enqueue` 调度到主线程。设计文档在 `AssetDeleteFolderTool.Execute` 伪码第 4 步又写了「通过 MainThreadQueue 调度到主线程」。如果工具自身再做一次调度，会导致双重入队；如果 Execute 内部不调度而是直接执行文件操作，那设计文档的描述具有误导性。需要明确工具的 Execute 方法在被调用时是否已经在主线程上下文中。

2. **`List.RemoveAt(0)` 扩容风险**：现有缓冲区用 `List<LogEntry>` + `RemoveAt(0)` 实现环形语义，这是 O(n) 操作。从 1000 扩到 5000 后，高频日志场景下（如循环打印）每秒可能数十次 O(5000) 的数组搬移。虽然这不在本次需求范围内，但既然扩容是本次需求，设计文档应至少标注此已知技术债。

3. **路径参数语义需要更精确的约定**：参数描述写「要删除的 Assets 子目录路径（相对于 Assets）」，但没有示例。用户可能传 `"Generated/Code"`（正确）或 `"Assets/Generated/Code"`（会变成 `Assets/Assets/Generated/Code`）。建议在参数 description 或 Error Handling 中加一个示例和对常见误用的处理。

4. **`console_getLogs` 返回值缺少 index 字段**：上下文模式依赖 `beforeIndex` 锚点索引，但返回的 LogEntry 没有 index 字段。调用方如何知道某条日志的 index 以便后续用 `beforeIndex` 定位？需要在返回模型中加入 `index` 字段，或在返回的 JSON 中包含 `bufferIndex` 等信息。

## 3. 改进建议

### 🔴 关键问题（必须修改）

1. **`console_getLogs` 返回值需包含 index 信息** — 上下文模式的核心交互流程是：先 getLogs 发现异常 → 取该条的 index → 用 beforeIndex 获取上下文。但当前 LogEntry 返回模型中没有 index 字段，调用方无法获取 index。建议在返回 JSON 中为每条日志增加 `"index"` 字段（即该条在 buffer 中的位置），或至少在非上下文模式的返回中附带 `startIndex`/`endIndex` 元数据。

2. **统一 Category 命名** — `ConsoleClearTool` 的 Category `"console"` 与现有 `ConsoleTool` 的 `"debug"` 不一致。方案二选一：(a) `ConsoleClearTool` 改为 `"debug"` 以匹配现有惯例；(b) 两个工具统一改为 `"console"` 并在需求文档中注明这是有意的 Breaking Change。同时需更新 requirements.md Req 2.4。

### 🟡 建议改进（推荐修改）

1. **明确 MainThreadQueue 调度职责** — 在 Architecture 或 Components 章节注明：「所有 IMcpTool.Execute 调用已由 JsonRpcDispatcher 通过 MainThreadQueue 在主线程执行，工具实现内部无需再做线程调度」。删除 `AssetDeleteFolderTool` 伪码第 4 步中的「通过 MainThreadQueue 调度到主线程」措辞，改为直接执行 `Directory.Delete` 和 `AssetDatabase.Refresh`。

2. **补充 Out of Scope 章节** — 建议在 Overview 后加一段，明确排除：仅支持删除目录（不支持删除单个文件）；keyword 为子串匹配（不支持正则）；不修改 ConsoleTool 的 Category 等。

3. **path 参数增加示例和归一化说明** — 在 `asset_deleteFolder` 参数模型的 description 中增加示例（如 `"Generated/Code"`），并在伪码中增加对 `Assets/` 前缀的自动剥离或明确拒绝的逻辑。

4. **标注 `List.RemoveAt(0)` 技术债** — 在 Architecture 的「关键约束」中增加一条：「当前缓冲区使用 `List<T>.RemoveAt(0)` 实现 FIFO 淘汰，扩容到 5000 后 O(n) 开销增大，后续可考虑改用 `LinkedList<T>` 或 circular buffer。本次不改动数据结构。」

### 🟢 锦上添花（可选优化）

1. **Overview 补充动机链接** — 在 Overview 中加一句引用 requirements.md 的核心 User Story，让读者无需切换文件即可理解「为什么」。

2. **属性测试增加 symlink 场景** — Property 1 的路径安全测试目前覆盖了 `..` 和绝对路径，建议补充 symlink 场景（如 Assets 下的 symlink 指向外部目录），这在 macOS/Linux 开发环境中是常见的。
