# 技术需求文档评审

对技术需求文档进行系统性评审：先按 Checklist 逐项检查，再给出自由评审意见和可落地的改进建议。

## 评审流程

1. **接收文档** — 用户提供技术需求文档的路径或内容
2. **通读全文** — 完整阅读文档，建立整体理解
3. **Checklist 逐项检查** — 按 6 大维度 19 个检查项逐一评估，每项给出 ✅ / ⚠️ / ❌ 和简要说明
4. **自由评审** — 跳出 Checklist，从整体视角给出额外发现和意见
5. **改进建议** — 针对发现的问题给出具体、可操作的修改建议
6. **生成报告** — 将评审结果写入 Markdown 文件，保存到工作区根目录下的 `.ai_review/` 目录

## Checklist 定义

### 1. 结构完整性

| # | 检查项 | 说明 |
|---|--------|------|
| 1.1 | 背景/目标/动机 | 是否清晰说明了为什么要做这个需求，要解决什么问题 |
| 1.2 | 功能范围（Scope） | 是否明确划定了本次需求要做的范围 |
| 1.3 | 边界（Out of Scope） | 是否明确说明了不做什么，避免需求蔓延 |
| 1.4 | 术语/缩写定义 | 如文档中使用了专有术语或缩写，是否有定义说明 |

### 2. 功能描述清晰度

| # | 检查项 | 说明 |
|---|--------|------|
| 2.1 | 无歧义性 | 功能描述是否清晰无歧义，不同读者能得出相同理解 |
| 2.2 | 输入/输出/行为 | 每个功能点是否有具体的输入、输出、行为描述 |
| 2.3 | 异常/边界场景 | 是否覆盖了异常流程和边界条件 |
| 2.4 | 优先级区分 | 是否区分了必须（Must）和可选（Nice to have） |

### 3. 技术可行性

| # | 检查项 | 说明 |
|---|--------|------|
| 3.1 | 架构兼容性 | 技术方案是否与现有架构/技术栈兼容 |
| 3.2 | 风险与依赖 | 是否识别了技术风险和外部依赖 |
| 3.3 | 性能指标 | 是否有量化的性能要求（延迟、吞吐、内存等） |
| 3.4 | 安全性 | 是否考虑了安全性、权限、数据隐私 |
| 3.5 | 框架影响 | 是否对当前项目框架造成根本性破坏，是否带来巨大的需求外连带影响风险 |
| 3.6 | 可扩展性 | 未来的扩展是否方便 |

### 4. 一致性与关联性

| # | 检查项 | 说明 |
|---|--------|------|
| 4.1 | 内部自洽 | 文档内部描述是否前后一致、不矛盾 |
| 4.2 | 外部一致 | 与已有文档、系统设计是否一致 |
| 4.3 | 接口引用 | 关联模块/接口是否有清晰引用 |

### 5. 可测试性

| # | 检查项 | 说明 |
|---|--------|------|
| 5.1 | 可转化为测试用例 | 需求描述是否具体到可以编写测试用例 |
| 5.2 | 验收标准 | 验收标准（Acceptance Criteria）是否明确 |

### 6. 可维护性与可读性

| # | 检查项 | 说明 |
|---|--------|------|
| 6.1 | 组织结构 | 文档组织是否清晰，章节层次是否合理 |
| 6.2 | 图表辅助 | 是否使用图表/流程图辅助说明复杂逻辑（如适用） |

## 评分标准

每个检查项使用三级评分：

- **✅ 通过** — 该项达标，无需改动
- **⚠️ 待改进** — 有内容但不够完善，需要补充或优化
- **❌ 缺失/不合格** — 完全缺失或存在严重问题

## 输出规范

1. **文件位置**：`<工作区根目录>/.ai_review/<文档名>-review-<YYYYMMDD-HHmmss>.md`
2. **文件格式**：UTF-8 编码，LF 换行，末尾留一空行
3. **报告结构**：读取 `references/report-template.md` 获取完整模板，按模板格式输出
4. **不在会话中展示完整报告**，只在会话中给出一句话总结（如「评审完成，发现 3 个待改进项和 1 个缺失项，报告已生成到 .ai_review/xxx.md」）

## 评审原则

### 上下文优先，拒绝机械评审

评审之前先理解文档的**定位和场景**。一份 MVP/实验性文档的评审标准和一份生产级系统设计不同。评审时反复问自己：

- 这个文档的定位是什么？（MVP？正式需求？原型验证？）
- 目标读者是谁？（团队内部技术人员？跨部门？外部合作方？）
- 实际使用场景是什么？（单用户工具？高并发服务？内部实验？）

**根据这些上下文调整评审的严格程度。** 不要对一个明确标注为"最小可行实验"的文档提出生产级高可用的要求。

### 具体规则

- **不说废话** — 报告中不要出现"整体质量很高""设计得很好""值得肯定"等恭维性语句。只输出问题、事实和建议
- **不吹毛求疵** — 如果某项在该文档的定位和场景下不适用或不重要，直接标 ✅ 并注明"该场景下无需关注"，而不是硬挑问题
- **术语判断要合理** — 目标读者群体内的公知术语（如技术团队内的 JSON-RPC、SSE、JWT）不算"术语缺失"。只有在文档使用了**项目特有的、非通用的**术语且未定义时，才标记为问题
- **贴合实际场景** — 评审意见必须结合文档描述的实际使用场景。例如一个本地单用户工具不需要考虑高并发；一个内部实验不需要完整的安全审计
- **具体可操作** — 每条改进建议都应该具体到能直接执行，而不是泛泛而谈
- **分清轻重缓急** — 关键问题和建议性改进要区分开，帮助作者聚焦
- **保持客观** — 如果某项无法判断（如缺乏项目上下文），标注为「无法判断」而非猜测
